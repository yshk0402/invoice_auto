# 外国人派遣事業者向け経費申請SaaS 要件定義（初版ドラフト）

## 0. 文書の位置づけ

- 本書は、外国人派遣事業者の毎月の経費申請作業負荷を軽減する社内SaaSの初期要件定義である。
- 本書には会話で確定した事実のみを記載し、未確定事項は `CONFIRM REQUIRED` として明示する。
- 本書は runtime API、DB スキーマ、インフラ構成、運用手順の詳細を定義しない。

## 1. 背景・目的

- 対象プロジェクトは、外国人派遣事業者の社内利用を前提とした経費申請SaaSである。
- 毎月の経費申請において、レシート回収、入力、確認、集計にかかる手作業負荷を下げることを目的とする。
- 初版では、ユーザーによるスマホ提出と、経理部による一覧確認・集計の成立を優先する。

## 2. v1 スコープ

### 2.1 対象

- 単一の派遣事業者向け社内SaaSとして利用する。
- 管理側は経理部を想定し、PC から Google スプレッドシートを主要な管理UIとして利用する。
- ユーザー側は事業者社員および派遣外国人を想定し、スマホから利用する。
- ユーザー側の基本フローは `Login` -> `Camera Scan` -> `OCR Correction / Submit` とする。
- 提出単位はレシート単位とする。
- OCR は入力補助として利用し、最終値はユーザーによる確認・修正を前提とする。
- レシート画像はクラウド保存を必須とする。
- 初期利用規模は数十人規模を想定する。

### 2.2 対象外

- 管理側による承認フロー
- 管理側からの差戻し・再提出フロー
- 行政機関や取引先など外部提出先への自動提出
- 複数事業者をまたぐテナント分離

## 3. 利用者と役割

| 役割 | 想定利用者 | 主な操作端末 | 主な責務 |
| --- | --- | --- | --- |
| `Admin` | 事業者の経理部 | PC | Google スプレッドシート上で一覧確認、集計、マスタ管理を行う |
| `Employee` | 事業者社員 | スマホ | レシート撮影、OCR結果の修正、送信を行う |
| `ForeignWorker` | 派遣外国人 | スマホ | レシート撮影、OCR結果の修正、送信を行う |

- ユーザー側ログイン方式は `社員ID + パスワード` とする。
- 管理側の主言語は日本語のみとする。
- ユーザー側は日本語、英語、中国語、ベトナム語に対応する。

## 4. 業務フロー

1. ユーザーがスマホでログインする。
2. ユーザーがレシートを撮影する。
3. システムがOCRでレシート情報を抽出する。
4. ユーザーが抽出結果を確認し、必要に応じて修正する。
5. ユーザーがレシート単位で送信する。
6. 管理側が Google スプレッドシートで提出内容を一覧確認し、月次集計に利用する。

## 5. 画面要件

### 5.1 管理側

- 管理側の主要UIは Google スプレッドシートとする。
- シート構造は `Master Sheet` と `Per-user Sheet Tab` を前提にする。
- `Master Sheet` はユーザー台帳とタブ対応表を管理する。
- `Per-user Sheet Tab` はユーザーごとの提出一覧を管理する。
- `Master Sheet` の列は `employee_id` `user_name` `role` `preferred_language` `sheet_tab_name` `active` とする。
- `Per-user Sheet Tab` の列は `submitted_at` `date` `merchant` `amount` `expense_category` `receipt_image` `submitter` `language` とする。
- `Master Sheet` のシート名は `Master Sheet` 固定とする。
- `Per-user Sheet Tab` の命名規則は `USER_<社員ID>` とする。

### 5.2 ユーザー側共通

- ユーザー側はスマホ操作を前提とする。
- 対応言語は日本語、英語、中国語、ベトナム語とする。
- 初期表示言語は端末言語を優先して決定する。
- 言語切替はログイン前後の両方で可能とする。
- ユーザー側の詳細な画面遷移、文言、エラーハンドリングは本書では未確定とし、`CONFIRM REQUIRED` で扱う。

### 5.3 `Login`

- 入力要素は `社員ID` と `パスワード` を前提とする。
- PoC 用の初期アカウントは 5 件を用意し、`id1` / `pass1` から `id5` / `pass5` までを利用する。
- アカウント発行は上記の固定アカウント配布を前提とする。
- ログイン画面では言語切替を可能とする。
- ログイン成功時は `Camera Scan` 画面へ遷移する。
- サインアップ導線、パスワード再設定導線は設けない。
- 認証失敗時の表示仕様、追加発行、再発行は未確定である。
- パスワード再設定フローは v1 では提供しない。

### 5.4 `Camera Scan`

- ユーザーはこの画面でレシートの写真を撮る。
- 1 回の提出対象はレシート 1 枚とする。
- 撮影後は `OCR Correction / Submit` 画面へ遷移する。
- この画面では管理機能や一覧確認機能を提供しない。
- OCR 実行前後の再撮影、画像アップロード失敗時の取り扱いは未確定である。

### 5.5 `OCR Correction / Submit`

- ユーザーはOCR抽出結果を確認し、必要に応じて修正する。
- 修正対象は `date` `amount` `merchant` `expense_category` とする。
- `expense_category` の選択肢は `transportation` `meal` `supplies` `lodging` `communication` `other` とする。
- 送信単位はレシート単位とし、1 回の送信で 1 件の提出データを作成する。
- 送信時には `receipt_image` `submitter` `submitted_at` `language` をあわせて提出データとして保持する。
- 送信時の完了表示、入力バリデーション、未入力時の扱いは未確定である。

### 5.6 画面遷移の最小契約

1. ユーザーは `Login` 画面で `社員ID` と `パスワード` を入力する。
2. ログイン成功後、`Camera Scan` 画面へ遷移する。
3. ユーザーはレシートを 1 枚撮影する。
4. 撮影後、`OCR Correction / Submit` 画面へ遷移する。
5. ユーザーは抽出結果を確認・修正し、レシート単位で送信する。

## 6. データ要件

### 6.1 管理構造

- 管理構造は `Master Sheet` と `Per-user Sheet Tab` を最小契約とする。
- ユーザーごとのタブを持つ運用を前提とする。
- `Master Sheet` はユーザー台帳、利用言語、役割、提出先タブ対応を保持する。
- `Per-user Sheet Tab` はユーザー単位の提出データ一覧を保持する。
- シート列定義は §5.1 の列構成を契約とする。
- タブ命名規則は `USER_<社員ID>` とする。

### 6.2 提出データ項目

初版で文書上の契約として扱う提出データ項目は以下とする。

| 項目 | 意味 |
| --- | --- |
| `date` | レシート日付 |
| `amount` | 金額 |
| `merchant` | 店名 |
| `expense_category` | 経費科目 |
| `receipt_image` | レシート画像 |
| `submitter` | 送信者 |
| `submitted_at` | 送信日時 |
| `language` | ユーザー画面の利用言語 |

- 上記以外の項目追加は、本書または別途合意された仕様で確定するまで行わない。
- 初版の `expense_category` 候補値は `transportation` `meal` `supplies` `lodging` `communication` `other` とする。
- 経費科目マスタの管理責任者は経理部とする。
- 経費科目の更新は経理部による `Master Sheet` の更新で行う。
- 経費科目ごとの必須・任意の区分や詳細バリデーションは未確定である。

### 6.3 管理側 Google スプレッドシート運用

1. 管理側は `Master Sheet` を開き、対象ユーザーの `employee_id` と `sheet_tab_name` を確認する。
2. 管理側は対応する `Per-user Sheet Tab` を開き、提出内容を一覧確認する。
3. 月次集計は Google スプレッドシート上で行う。
4. 初版では、管理側専用Web画面や承認画面は提供しない。

### 6.4 多言語文言の最小契約

- ユーザー向け固定文言は日本語、英語、中国語、ベトナム語の 4 言語をそろえる。
- 表示言語は端末言語またはユーザーが選択した言語に従う。
- 管理側向け文言は日本語のみを対象とする。
- 翻訳文言の管理方法と翻訳更新フローの詳細は未確定である。

## 7. 非機能・運用前提

- 管理側は PC 利用、ユーザー側はスマホ利用を前提とする。
- OCR は自動確定ではなく、人による最終確認を前提とする。
- レシート画像はクラウド保存必須とする。
- レシート画像の保存先は Google Drive 内の所定フォルダとする。
- PoC 期間中は画像保持期間を定めない。
- 画像閲覧権限は、ユーザーが共有する Google Drive マイドライブ配下フォルダの共有権限に従う。
- 監査ログ、セキュリティ要件の詳細は未確定である。

## 8. 成功指標

- 初版の主指標は `提出完了率向上` とする。
- 比較対象となる現状値、目標値、測定期間、測定方法は未確定である。

## 11. 技術設計・PoC準備の観点

### 11.1 OCR 補助の設計観点

- OCR は `date` `amount` `merchant` `expense_category` の入力補助に限定する。
- OCR の抽出結果は確定値として扱わず、必ず `OCR Correction / Submit` 画面で人が確認する前提とする。
- OCR ベンダー選定、精度基準、失敗時再実行方針は `CR-07` の解消まで決め打ちしない。
- 実装時は、OCR 不成功でも画像保持と手入力への遷移を妨げない構成を優先する。

### 11.2 レシート画像保存の設計観点

- アプリケーションが保持する提出データ契約には、画像本体ではなく `receipt_image` として参照情報を保持する。
- 画像保存先は Google Drive 内の所定フォルダとし、他ストレージベンダーは v1 では扱わない。
- フォルダID、共有設定、接続方式の実装詳細は、実際に共有される Google Drive フォルダを前提に決定する。
- PoC では画像保持期間を設けないが、削除運用やアーカイブ運用は本書の対象外とする。

### 11.3 Google スプレッドシート連携の設計境界

- Google スプレッドシートは管理側の主要UIであり、初版では管理側専用Web画面で代替しない。
- 書き込み先は `Master Sheet` と `Per-user Sheet Tab` の契約列に限定する。
- `Master Sheet` の責務はユーザー台帳とタブ対応管理とし、提出データ本体は `Per-user Sheet Tab` に記録する。
- 連携実装では、列追加やシート名変更を前提にした可変仕様を持ち込まない。

### 11.4 認証・アカウント運用の設計観点

- 認証方式は `社員ID + パスワード` に固定する。
- PoC では `id1` / `pass1` から `id5` / `pass5` の固定アカウントだけを対象にする。
- サインアップ、パスワード再設定、セルフサービスのアカウント管理は v1 対象外とする。
- 追加発行や再発行の運用は未確定のため、初版実装では管理画面や申請導線を持ち込まない。

### 11.5 監査ログとアクセス管理の設計観点

- 監査ログの対象イベント、保存期間、閲覧権限は `CR-05` の解消まで未確定とする。
- 初版で技術的に発生しうるイベントがあっても、監査ログを要件確定前に契約機能として実装しない。
- アクセス管理は Google Drive 共有権限と Google スプレッドシートの閲覧権限に依存する前提とし、独自の詳細権限制御は v1 で追加しない。
