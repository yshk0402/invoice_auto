# AGENTS.md

## 0. 目的

- このリポジトリでは、要件を勝手に補完せず、確認済みの事実だけで開発を進める。
- 仕様の唯一ソースは [REQUIREMENTS.md](REQUIREMENTS.md) とする。
- [TASK.md](TASK.md) は要件確定を進めるためのバックログ、`AGENTS.md` はその運用ルールを定義する。

## 1. 最優先原則

1. 要件補完禁止
   - ユーザーが明示していない内容は埋めない。
   - 不足情報は推測で埋めず、`CONFIRM REQUIRED` として扱う。
2. 未確定事項は必ず質問
   - 実装や設計判断に影響する未確定事項に遭遇したら、作業を進める前にユーザーへ確認する。
   - `REQUIREMENTS.md` に未登録の不明点を見つけた場合は、ユーザー確認の対象に追加し、未解消なら `CONFIRM REQUIRED` として反映する。
3. 仕様変更時は `REQUIREMENTS.md` 先更新
   - 仕様変更、スコープ変更、用語変更、役割変更は、コードやタスクより先に `REQUIREMENTS.md` を更新する。
4. v1 範囲外の機能追加禁止
   - 承認、差戻し、外部提出、自動テナント分離など、現行 v1 対象外の機能は追加しない。
5. Google スプレッドシート前提と多言語スマホUXを壊さない
   - 管理側の主要UIは Google スプレッドシート前提とする。
   - ユーザー側はスマホ操作と 4 言語対応を前提に扱う。
6. `TASK.md` は必ず要件にトレースさせる
   - すべてのタスクは `REQUIREMENTS.md` の節番号または `CR-*` に対応づける。
   - 対応先を持たないタスクは追加しない。

## 2. 作業順序

1. `REQUIREMENTS.md` を読む。
2. 対象の変更が既存要件のどこに紐づくかを確認する。
3. 要件が未確定ならユーザーへ確認する。
4. ユーザー確認後、必要なら `REQUIREMENTS.md` を更新する。
5. その変更に応じて `TASK.md` と本書を同期する。
6. 要件が確定した後にのみ設計・実装へ進む。

## 3. 文書運用ルール

- 未確定事項は必ず `CONFIRM REQUIRED` と明記する。
- `TODO` や曖昧な仮置き文言で済ませない。
- `REQUIREMENTS.md` の意思決定ログは削除せず、日付付きで追記する。
- 新しい未確定事項を見つけた場合は、誰に確認するかが分かる形で追加する。
- `TASK.md` のタスクは、完了条件とトレース先が分かる表現にする。

## 4. 実装ガードレール

- runtime API、DB スキーマ、保存先ベンダー、認証基盤、OCR サービスは、要件未確定のまま決め打ちしない。
- 要件で確定していない入力項目、権限、保存期間、バリデーション、業務締め処理は実装しない。
- `Employee` と `ForeignWorker` に差分を設ける場合は、事前に要件確認を取る。
- 画面設計時は、スマホUIの可読性と多言語展開を優先し、日本語固定前提で組まない。
- 管理向け補助UIを追加する場合でも、Google スプレッドシート運用を置き換える前提で設計しない。

## 5. タスク更新ルール

- 新規タスクには `T-xxxx` 形式のIDを振る。
- 各タスクに `Trace:` を付けて、`REQUIREMENTS.md` の節番号または `CR-*` を記載する。
- `CR-*` が解消されたら、対応するタスクを更新し、必要なら意思決定ログへ反映する。
- 要件未確定のまま設計・実装タスクへ進む状態を残さない。

## 6. このリポジトリで守る前提

- 管理側: PC + 日本語 + Google スプレッドシート
- ユーザー側: スマホ + 日本語 / 英語 / 中国語 / ベトナム語
- ユーザー認証: `社員ID + パスワード`
- 初版送信単位: レシート単位
- OCR: 補助機能。最終確定は人が行う
- 初版主指標: 提出完了率向上

## 7. 変更前チェック

- この変更は `REQUIREMENTS.md` のどの節に対応しているか。
- この変更は既存の `CR-*` を解消するためのものか。
- 未確定事項を推測で埋めていないか。
- v1 範囲外の機能を持ち込んでいないか。
- `TASK.md` に対応タスクがあるか、なければ追加すべきか。
